# Rigoblock SmartPool - Cyfrin Solskill security audit

**Target:** Rigoblock SmartPool (ERC-1967 proxy-based pool/fund management protocol)
- Proxy: `0xEfa4bDf566aE50537A507863612638680420645C` (Solidity 0.8.17)
- Implementation: `0x1DB955265B8DC18715cAb12E805F9b71Fa545420` (Solidity 0.8.28)
- 45 Solidity source files, ~3,143 lines

**Methodology:** Cyfrin solskill production-grade Solidity development and security standards

## [HIGH] Pool owner is a single EOA -- no multisig governance protection

- **Severity:** HIGH
- **File:** `/protocol/core/actions/MixinOwnerActions.sol:42-45`
- **Description:** The pool owner (`0xcA9F5049c1Ea8FC78574f94B7Cf5bE5fEE354C31`) is an externally owned account (EOA). All critical owner functions -- `setOwner`, `setTransactionFee`, `changeSpread`, `changeMinPeriod`, `changeFeeCollector`, `setKycProvider`, `setAcceptableMintToken`, `purgeInactiveTokensAndApps` -- are protected only by the `onlyOwner` modifier, which checks against a single stored address. Cyfrin solskill standard #12 explicitly states: "the admin must be a multisig from the very first deployment -- never use the deployer EOA as admin." The Trail of Bits reference on "Maturing Your Smart Contracts Beyond Private Key Risk" emphasises that single-EOA governance is no longer acceptable for DeFi protocols.
- **Impact:** If the owner's private key is compromised, the attacker gains full control over the pool. They can: set the transaction fee to the maximum 1% and redirect fees to their own address, change the spread to 5% (the maximum) to extract value from all minters/burners, set a malicious KYC provider that blocks legitimate users, transfer ownership permanently, accept arbitrary tokens to manipulate NAV calculations, and purge active tokens to break pool accounting. Additionally, `setOwner` uses a single-step transfer pattern (not `Ownable2Step`), meaning a typo in the new owner address would permanently lock all owner functionality with no recovery path.
- **Recommendation:** Replace the EOA owner with a multisig (e.g. Safe) before any further funds are deposited. Implement a two-step ownership transfer pattern (`Ownable2Step`) so that the new owner must explicitly accept ownership. This prevents accidental or malicious single-transaction ownership transfers to uncontrolled addresses.

## [HIGH] Unrestricted arbitrary storage read exposes all internal state including private mappings

- **Severity:** HIGH
- **File:** `/protocol/core/state/MixinStorageAccessible.sol:10-37`
- **Description:** `getStorageAt(uint256 offset, uint256 length)` and `getStorageSlotsAt(uint256[] memory slots)` are `public view` functions that allow anyone to read arbitrary storage slots from the proxy contract. While storage is technically readable on-chain via `eth_getStorageAt`, these helper functions make it trivially easy for attackers to compute and batch-read derived storage slots (e.g. user account mappings, operator approval mappings, internal fee collector addresses) without understanding Solidity's storage layout. Critically, `getStorageAt` takes an `offset` and `length` parameter and reads sequential 32-byte words, enabling enumeration of entire storage regions. The `slots` variant takes an arbitrary array allowing targeted reads of computed mapping slots.
- **Impact:** Attackers can efficiently extract all user balances, activation times, operator approval states, and pool parameters in a single call. This information leakage enables targeted attacks: an attacker can identify high-value accounts, determine exact lock-up expiry times (to time sandwich attacks at the moment of unlock), and discover operator relationships. While the data is technically available on-chain, these convenience functions dramatically lower the barrier to exploitation. More importantly, for a protocol managing user funds, exposing this level of introspection without any access control violates the principle of defence in depth.
- **Recommendation:** If these functions are only needed for off-chain tooling or specific extensions, restrict access with an `onlyOwner` modifier or limit the readable range to specific known storage slots. If the functions must remain public, document the security implications and ensure all sensitive operations are protected against information-assisted attacks (e.g. front-running based on known activation times).

## [HIGH] ERC-20 transfer/transferFrom/approve silently return false instead of reverting, enabling phantom token approvals and transfers

- **Severity:** HIGH
- **File:** `/protocol/core/sys/MixinAbstract.sol:7-19`
- **Description:** The `MixinAbstract` contract implements ERC-20 methods `transfer`, `transferFrom`, and `approve` as empty function bodies that return the default value `false` (the default for `bool` in Solidity). These are not marked as reverting, not marked with `revert`, and have no body at all. Any caller invoking `transfer(to, value)` or `approve(spender, value)` on a pool token will receive `success = false` silently. The `allowance` function always returns `0`. While this appears intentional (pool tokens are non-transferable by design, with internal balance tracking), the implementation violates the ERC-20 standard in a dangerous way: compliant contracts checking `require(token.transfer(...))` will correctly fail, but contracts that check `if (token.transfer(...))` and have a fallback path will silently take the wrong branch.
- **Impact:** Any protocol or contract that integrates pool tokens and relies on standard ERC-20 behaviour will silently fail. DeFi composability is broken: tokens cannot be used as collateral, in DEX pools, or in any standard DeFi protocol that expects ERC-20 compliance. More critically, contracts that call `approve` followed by `transferFrom` on behalf of users will believe the approval succeeded when it did not, potentially leading to stuck funds or incorrect accounting in integrating protocols. The silent failure mode is particularly dangerous because it does not revert, making it difficult to detect during testing.
- **Recommendation:** All non-implemented ERC-20 methods should explicitly `revert` with a descriptive custom error (e.g., `error PoolTokenNonTransferable()`) rather than silently returning `false`. This prevents any integrating contract from mistakenly believing an operation succeeded. This is the defensive coding approach mandated by Cyfrin standards: "Everything will be attacked -- assume that any code you write will be attacked and write it defensively."

## [MEDIUM] Unsafe downcast of mintedAmount from uint256 to uint208 can silently truncate large balances

- **Severity:** MEDIUM
- **File:** `/protocol/core/actions/MixinActions.sol:200,206`
- **Description:** In `_allocateMintTokens`, the `mintedAmount` (a `uint256`) is added to `userBalance` using `uint208(feePool)` (line 200) and `uint208(mintedAmount)` (line 206). The `UserAccount.userBalance` field is defined as `uint208` in the `ISmartPoolState` interface. The cast from `uint256` to `uint208` is performed using a raw Solidity cast without SafeCast. While the contract imports `SafeCast` and uses `SafeCast for uint256` in `MixinActions`, the `.toUint208()` method is not used here -- instead a raw `uint208()` cast is used. If `mintedAmount` ever exceeds `type(uint208).max` (approximately 4.1e62), the value would silently truncate. Similarly, in `_allocateBurnTokens` (line 270), `uint208(amountIn)` is used for subtraction.
- **Impact:** While `uint208` is very large and practical overflow is unlikely under normal conditions, the use of unsafe casts instead of `SafeCast.toUint208()` is inconsistent with the contract's own pattern of using SafeCast elsewhere. If a pool token with very small decimals were somehow created (despite the 6-decimal minimum), or if the unitary value dropped to extremely low values, the mathematical product `(amountIn * 10 ** components.decimals) / components.unitaryValue` could theoretically produce values above `uint208.max`. The silent truncation would result in a user receiving far fewer tokens than they paid for, with the difference effectively stolen.
- **Recommendation:** Replace all raw `uint208()` casts with `SafeCast.toUint208()` to guarantee that any overflow is caught with a revert rather than silently truncated. The contract already imports SafeCast, so this is a minimal change.

## [MEDIUM] The updateUnitaryValue function lacks reentrancy protection and can be called by anyone to manipulate NAV timing

- **Severity:** MEDIUM
- **File:** `/protocol/core/actions/MixinActions.sol:80-90`
- **Description:** The `updateUnitaryValue()` function is `external` with no access control and no `nonReentrant` modifier. The comment says "Reentrancy protection provided by calling functions (mint, burn, depositV3, donate)" but `updateUnitaryValue` is itself a public entry point that can be called directly. It calls `_updateNav()` which performs external calls to `IEOracle` and `IEApps` (via `_computeTotalPoolValue`), reads token balances, and then writes to `poolTokens().unitaryValue` in storage. An attacker can call `updateUnitaryValue()` at a strategically chosen time (e.g. during flash loan manipulation of Uniswap pool prices used by the oracle) to store a manipulated NAV value. Because the function writes to storage (`poolTokens().unitaryValue`), this stale/manipulated NAV will persist and affect subsequent mint/burn calculations.
- **Impact:** An attacker can manipulate the stored unitary value by calling `updateUnitaryValue()` during a flash loan that temporarily inflates or deflates token prices visible to the oracle. Even if the price manipulation is transient, the stored NAV update persists. A subsequent `mint` at an inflated NAV means the minter receives fewer pool tokens than deserved, while a subsequent `burn` at an inflated NAV means the burner receives more base tokens than deserved. This creates an MEV-extractable sandwich: (1) flash loan to manipulate oracle prices, (2) call `updateUnitaryValue()` to store manipulated NAV, (3) immediately mint at favourable price, (4) wait for NAV to normalise, (5) burn at true price.
- **Recommendation:** Add the `nonReentrant` modifier to `updateUnitaryValue()`. Consider adding access control (e.g. `onlyOwner`) or a time-based rate limit (e.g. minimum interval between NAV updates) to prevent manipulation. If the function must remain permissionless, add oracle price sanity checks or TWAP-based smoothing to prevent single-block price manipulation from affecting the stored NAV.

## [MEDIUM] Proxy constructor uses hardcoded function selector and weak initialisation failure check

- **Severity:** MEDIUM
- **File:** `/proxy/src/Contract.sol:104-107`
- **Description:** The proxy constructor calls the implementation's `initializePool` using a hardcoded selector `0x250e6de0` rather than `IRigoblockPool.initializePool.selector`. The initialisation success check relies on `require(returnData.length == 0, "POOL_INITIALIZATION_FAILED_ERROR")` -- meaning it only verifies that the return data has zero length rather than checking the boolean return of `delegatecall` (stored in the unnamed first return variable). The `delegatecall` result is captured as `(, bytes memory returnData)` where the first (bool success) value is discarded. If the `delegatecall` returns `success = false` with zero-length return data (which happens when the called function runs out of gas or calls a non-existent function on a contract with no fallback), the proxy would be deployed with an uninitialised pool.
- **Impact:** If the implementation contract changes and the `initializePool` selector changes (which is unlikely given the ABI stability, but possible during upgrades), the call would go to the wrong function or to the fallback. If the fallback returns empty data (which the SmartPool fallback could do for unmapped selectors after the `staticcall` path), the pool proxy would deploy in an uninitialised state. An uninitialised pool would have a zero owner, allowing anyone who finds it to race to call `initializePool` directly. Additionally, using a string error message (`"POOL_INITIALIZATION_FAILED_ERROR"`) instead of a custom error contradicts Cyfrin standard #2 (prefer `revert` with custom errors).
- **Recommendation:** Check the `bool success` return value from `delegatecall`. Use `IRigoblockPoolProxyFactory.initializePool.selector` or the interface's computed selector rather than a hardcoded hex value. Replace the `require` with a `revert` using a custom error. Note: the proxy is at Solidity 0.8.17 where some of these patterns were less established, but for a production mainnet deployment this should be upgraded.

## [MEDIUM] Owner can set transaction fee and spread that immediately affects existing holders without time-lock

- **Severity:** MEDIUM
- **File:** `/protocol/core/actions/MixinOwnerActions.sol:58-75,168-173`
- **Description:** The functions `setTransactionFee`, `changeSpread`, and `changeMinPeriod` take effect immediately when called by the owner with no time-lock, no governance delay, and no notification to existing pool holders. The transaction fee can be set up to 100 basis points (1%) and the spread up to 500 basis points (5%). These parameters directly affect the economics of every mint and burn operation. A pool operator could front-run a large pending burn by increasing the spread to the maximum, extracting significantly more value from the burning user than expected. Because the owner is a single EOA (see finding #1), this action can be executed in a single transaction with no warning.
- **Impact:** Consider a holder burning tokens worth 100,000 USDC with the current spread of 10 bps (0.1%). They expect to lose ~100 USDC to spread. The owner front-runs their burn transaction by setting the spread to 500 bps (5%), causing the holder to lose 5,000 USDC to spread instead -- a 50x increase. The owner can then reset the spread in the next block. Similarly, setting the transaction fee to 1% would extract 1% of every mint/burn in pool tokens to the fee collector. The lack of a time-lock means holders cannot monitor and react to parameter changes.
- **Recommendation:** Implement a time-lock pattern (e.g. 24-48 hours) for fee and spread changes, emitting a pending-change event that holders can observe before the change takes effect. At minimum, use a governance multisig (see finding #1) to reduce the risk of unilateral parameter manipulation.

## [MEDIUM] Missing reentrancy protection on fallback delegatecall path allows extensions to re-enter pool state

- **Severity:** MEDIUM
- **File:** `/protocol/core/sys/MixinFallback.sol:28-68`
- **Description:** The `fallback()` function routes unrecognised selectors to either extension contracts (via `_extensionsMap`) or adapter contracts (via `IAuthority.getApplicationAdapter`). When `shouldDelegatecall` is true (which happens for all extension calls and for adapter calls when `msg.sender == pool().owner`), the call is executed via `delegatecall` in the pool's context with no reentrancy guard. The `nonReentrant` modifier from `ReentrancyGuardTransient` only protects `mint`, `burn`, `mintWithToken`, and `burnForToken`. Extension and adapter delegatecalls can modify pool storage and call back into the pool's mint/burn functions. While the extensions are presumably trusted (managed by governance), adapters are approved by the Authority contract and could be added by whitelisters -- a broader trust set.
- **Impact:** A malicious or compromised adapter contract approved by a whitelister could re-enter the pool during a delegatecall to manipulate state between operations. For example, an adapter could call `updateUnitaryValue()` during execution to set a manipulated NAV, then trigger a mint or burn at the wrong price. The transient reentrancy guard only prevents re-entering `nonReentrant`-protected functions, but other state-modifying functions (like `setOperator`, owner actions, or other adapter calls) remain callable during reentrancy.
- **Recommendation:** Add reentrancy protection to the fallback function itself, or ensure that all state-modifying delegatecall targets are individually protected. Consider implementing a global reentrancy lock that covers all entry points, not just mint/burn.

## [MEDIUM] Fee collector balance accumulation uses unsafe uint208 addition with no overflow check

- **Severity:** MEDIUM
- **File:** `/protocol/core/actions/MixinActions.sol:200,283`
- **Description:** In both `_allocateMintTokens` (line 200) and `_allocateBurnTokens` (line 283), the fee collector's balance is incremented using direct addition: `accounts().userAccounts[feeCollector].userBalance += uint208(feePool)`. This operation has two unsafe elements: (1) the `uint208()` cast truncates silently (see finding on unsafe downcasting), and (2) the `+=` operation on a `uint208` field does not check for overflow. While Solidity 0.8.x has built-in overflow checks for standard arithmetic, the addition here is between a `uint208` storage variable and a `uint208` value. The checked addition will revert on overflow, but the truncating cast before addition means the added value may be incorrect. The fee collector's balance grows monotonically (fees are never withdrawn through the burn mechanism in the normal path), meaning it accumulates over the lifetime of the pool.
- **Impact:** If the fee collector never burns their pool tokens, their balance continuously grows. While uint208 is extremely large, the combination of an unsafe cast (potentially adding a truncated smaller value) and monotonic growth creates a latent risk. More practically, in `_allocateBurnTokens`, the fee collector's activation time is set to `uint48(block.timestamp + 1)` -- meaning fee tokens have a trivially short lock-up of 1 second, which is inconsistent with the pool's stated minimum lock-up period of 1-30 days.
- **Recommendation:** Use `SafeCast.toUint208()` for all casts. Consider whether the fee collector's activation time should match the pool's `minPeriod` rather than being hardcoded to `block.timestamp + 1`.

## [MEDIUM] Proxy implementation is immutable per-proxy with no upgrade mechanism in the proxy itself

- **Severity:** MEDIUM
- **File:** `/proxy/src/Contract.sol:92-100`
- **Description:** The proxy contract stores the implementation address in the EIP-1967 implementation slot during construction and provides no function to update it afterward. The only way to change the implementation is if the implementation contract itself (via a delegatecall through the fallback) writes to the implementation slot. The `MixinFallback.sol` does route to an `eUpgrade` extension via the `_extensionsMap`, suggesting an upgrade mechanism exists externally, but the proxy has no explicit upgrade function and no upgrade event emission mechanism beyond the constructor. The implementation slot write is done via a struct-based storage access pattern (`getImplementation().implementation = implementation`) rather than the standard EIP-1967 assembly pattern. If the `eUpgrade` extension is misconfigured, removed, or the ExtensionsMap is changed to not map upgrade selectors, the proxy becomes permanently non-upgradeable.
- **Impact:** If the upgrade extension is ever removed from the ExtensionsMap, or if the governance authority removes the upgrade adapter, the pool proxy becomes permanently locked to its current implementation. Any critical bug discovered in the implementation would be unfixable. The lack of an explicit upgrade function in the proxy itself means the upgrade path depends entirely on the correct configuration of external contracts (ExtensionsMap and Authority).
- **Recommendation:** Consider adding an explicit `upgradeTo(address)` function in the proxy itself, protected by a governance mechanism, as a fallback upgrade path independent of the extension system. This provides defence in depth against extension misconfiguration.

## [MEDIUM] Inconsistent token activation tracking between acceptedTokensSet and activeTokensSet creates potential NAV manipulation window

- **Severity:** MEDIUM
- **File:** `/protocol/core/actions/MixinOwnerActions.sol:140-154` and `/protocol/core/actions/MixinActions.sol:155-157`
- **Description:** The system uses two separate token tracking sets: `acceptedTokensSet` (what the owner has approved for minting) and `activeTokensSet` (what tokens the pool actually holds and includes in NAV calculations). When an owner calls `setAcceptableMintToken(token, true)`, the token is added only to `acceptedTokensSet`. It is not added to `activeTokensSet` until the first `_mint` call with that token (line 157 of MixinActions.sol). This two-phase approach was designed to prevent an attack vector documented in the code comment on line 148 of MixinOwnerActions.sol: "accept token -> purge (removes from active) -> mint (NAV drops)". However, there is a window between when `addUnique` is called in `_mint` (adding to `activeTokensSet`) and when the NAV is recalculated with the new token's oracle price. The `_updateNav()` is called at line 132 of `_mint` before the token is added to `activeTokensSet` at line 157, meaning the NAV used for minting does not include the newly added token's value, even though the token transfer has already occurred.
- **Impact:** When a user mints with a non-base token for the first time, the NAV calculation at line 132 does not include the new token's value (because it is not yet in `activeTokensSet`). The token is transferred at lines 151-152, then added to `activeTokensSet` at line 157, then the mint amount is calculated using the old NAV at line 168. The transferred token value is effectively "invisible" to the NAV calculation used for this mint. This means the minter receives pool tokens priced at a NAV that does not account for the tokens they just deposited, which could result in receiving slightly more or fewer pool tokens than correct depending on the oracle conversion at line 163-165. The first mint with any new token type will have a subtly incorrect price.
- **Recommendation:** Move the `activeTokensSet().addUnique()` call to before `_updateNav()`, or restructure the mint flow so that NAV recalculation happens after all token transfers and set updates are complete. Alternatively, ensure the `_updateNav` calculation accounts for the incoming transfer by including it explicitly.
